# Contributing

Thanks for contributing to MoQ (Media over QUIC). This is a polyglot monorepo:
Rust crates under `rs/`, TypeScript/JavaScript packages under `js/`, language
wrappers under `py/`, `swift/`, `kt/`, `go/`, and docs under `doc/`. The
architecture overview and per-area conventions live in the `CLAUDE.md` files at
the repo root and inside each directory. Read the one next to the code you are
touching before you start.

## Branch targeting (read this first)

There are two long-lived branches. The split is about **semver breakage, not
size**:

- **`dev`** is the default integration branch. **Base your work on `dev` and open
  your PR against `dev` unless a maintainer tells you otherwise.** All
  wire-protocol changes (anything under `rs/moq-net`, including `moq-lite` /
  `moq-transport` framing) and any breaking public-API change (a renamed, removed,
  or signature-changed `pub` / exported item in `rs/moq-ffi`, `rs/libmoq`,
  `rs/moq-net`, `rs/hang`, `js/net`, `js/hang`, or the language wrappers) must go
  here. Catalog/container format changes that alter existing on-the-wire framing
  belong here too.
- **`main`** carries additive, non-breaking work only: bug fixes, new behavior,
  new/additive APIs, docs, and refactors that preserve the existing public and
  wire contracts.

When in doubt, target `dev`; reviewers will redirect you if a change turns out to
be purely additive. Both branches run the same CI on pull requests, so no extra
setup is needed either way.

When you create a working branch, base it on the **freshly fetched remote**
(`git fetch origin` first, then branch off `origin/dev`), not on a local `dev`
that may lag behind.

## Getting started

The reproducible toolchain comes from the Nix dev shell, which matches CI:

```bash
nix develop --command just check   # run all tests and lints
nix develop --command just fix     # auto-format and auto-fix lint
nix develop --command just build   # build everything
```

`just` is the task runner (see the root and per-directory `justfile`s). If Nix is
unavailable, use `cargo` (Rust) or `bun` (JS) directly. For JS work you can scope
to that workspace with `just js check`, `just js fix`, and `just js test`. Use
`bun` for all JS tooling, never npm/yarn/pnpm.

## Making a change

1. Pick the base branch per **Branch targeting** above and branch off the fresh
   remote.
2. Make your change.
3. Run `just fix` to format and auto-fix, then `just check` to verify tests and
   lints pass.
4. Walk the **Cross-Package Sync** table in the root `CLAUDE.md`: a change in one
   area (e.g. `rs/moq-ffi`, `rs/moq-net`, `rs/hang`) usually needs matching
   updates in the paired packages, language bindings, and docs. Update them in the
   same PR, or say why you skipped a row in the PR description.
5. Add tests where they are easy to write (unit tests live next to the code as
   `*.test.ts` in JS and `#[cfg(test)]` modules in Rust).
6. Keep the PR title and description accurate as the change evolves. The
   description becomes the squash-merge commit message.

## Code style

- **No em dashes** in code, comments, commit messages, or any prose. Use a period
  and a new sentence, or a comma/parenthesis.
- **Document every exported symbol.** Each `pub` Rust item and each exported JS/TS
  symbol gets a doc comment, plus a module-level doc on each entrypoint. These
  render on the published docs, so a missing one is a hole a consumer hits.
- Comment the non-obvious **why** (a constraint, an invariant, a workaround), not
  the obvious what. Comments describe the current state of the code, not its
  history.
- **API design is the thing to get right.** Prefer the smallest, hardest-to-misuse
  public surface; take an options struct/object rather than positional parameters
  when a function might gain knobs later. Let the type system make misuse
  unrepresentable.
- **No AI attribution in source.** Do not add `Co-Authored-By` trailers or
  "generated by" markers for any AI/LLM to commits, and do not tag code or doc
  comments. Authorship is yours.

## Reviews

PRs are reviewed before merge. Push the clear, unambiguous fixes from review
feedback directly, and raise anything architectural or open to interpretation as a
comment rather than guessing.
